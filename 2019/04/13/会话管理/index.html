网络通信协议

#会话管理

http每次请求都是握手/挥手,第二次和第一次请求时没有任何关联的,属于无状态协议,那么问题来了,既然http是无状态协议,那服务器是怎么区分不同的用户的呢?,这就是会话了

Cookie

Cookie是由web服务器保存在用户浏览器上的小文件，它可以包含用户的信息，无论何时用户链接到服务器，web站点都可以访问Cookie信息。

Cookie可以是临时的，也可以是持续的

临时的Cookie只在浏览器中保存一段规定的时间，一旦超过规定时间，该Cookie就会被系统清除。

持续的Cookie则保存在用户的Cookie文件中，下一次用户返回时，仍然可以对它进行调用，在Cookie文件中保存Cookie，有些用户担心Cookie的用户信息被窃取。其实网站以外的用户无法跨过网站来获取Cookie信息 。

存储

cookie存储在浏览器端的，不同的浏览器存储的cookie数量和数据的大小都不一致。大多数情况下单个域名限制最多保存20个cookie，每个cookie保存的数据不能超过4K

安全

服务端安全

cookie是存储在用户端的,可以被用户修改,所以服务端不能直接通过一个cookie来确定用户身份,需要用一定的方式加密或者对等存储(cookie作为凭证,在服务端记录对应数据),服务端session就是使用这种方法存储的

用户端安全

用户端的cookie安全的,网站以外的用户无法跨过网站来获取用户的cookie信息,但是有心之人可能会通过ajax方法,让用户访问A网站,却使用B网站的脚本进行敏感操作.(详细内容可自行搜索”跨站点脚本攻击”)

SESSION

Session在网络应用在称为“会话控制”，Session 对象存储特定用户会话所需的属性及配置信息。这样，当用户在应用程序的 Web 页之间跳转时，存储在 Session 对象中的变量将不会丢失，而是在整个用户会话中一直存在下去。

Session是另一种记录客户状态的机制，不同的是Cookie保存在客户端浏览器中，而Session保存在服务器上。客户端浏览器访问服务器的时候，服务器把客户端信息以某种形式记录在服务器上。这就是Session。

原理

  1. 用户A第一次进入，没有附带任何标识信息(通常是cookie)
  2. 服务端接收请求,给与用户A一个会话标识(通常是set_cookie,cookie值将加密)
  3. 服务端根据会话标识,在服务器本地存储用户信息
  4. 用户A端获取到会话标识,存储到用户端本地
  5. 用户A第二次请求,附带会话标识(通常是cookie)
  6. 服务端通过会话标识,找到服务端相应的用户信息

在没有Cookie的情况下，Session使用方法

通过其他方式传递sessionid，来实现Session的使用

  1. 设置php.ini中的session.use_trans_sid = 1或者编译时打开打开了–enable-trans-sid选项
  2. 让PHP自动跨页传递session id
  3. 手动通过URL传值、隐藏表单传递session id
  4. 用文件、数据库等形式保存session_id,在跨页过程中手动调用，实现Session共享

COOKIE与SESSION的区别

  1. cookie 是存放在浏览器端，不同的浏览器存储的cookie数量和数据的大小都不一致。大多数情况下单个域名限制最多保存20个cookie，每个cookie保存的数据不能超过4K。
  2. session存储在服务端，默认是以文件的形式存储，也可以存储在数据库和redis、memcache等缓存内存中。
  3. session是占用的服务器内存，所以内存越大，能存的值就越大，原则上讲无上限，一般用于存储对安全要求较高的重要数据;

API/Token

Token其实和session原理差不多,服务端通过给用户发送一个token,用户通过该token进行请求服务端,这种会话验证方式一般用于跨平台开发,以及接口开发,大概步骤为:

  1. 用户A第一次进入,通过验证机制(账号密码登陆)请求服务端token
  2. 服务端验证成功,给用户发送一个token(针对用户)
  3. 服务端根据token,在服务端存储对应的数据(文件,mysql,redis等)
  4. 用户A端获取到token,存储到用户端本地
  5. 用户A请求某接口,带上token
  6. 服务端通过token,验证用户有效性,返回数据
